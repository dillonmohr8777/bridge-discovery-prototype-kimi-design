# Kimi design brief: Bridge

## Mission

Create a substantially better visual and interaction design for Bridge, a browser-based cannabis-industry directory and professional network for brands, dispensaries, retailers, sales representatives, and administrators.

This is an isolated design lab copied from `dillonmohr8777/bridge-discovery-prototype`. Work here first. The original repository and any real or future production app must remain untouched.

## What you may do in this design lab

- Redesign the UI, information architecture, responsive layout, components, CSS tokens, motion, empty states, loading states, error states, and interaction details.
- Add realistic but clearly fictional fixtures and polished flows for the existing roles and routes.
- Make strong design decisions independently and iterate without waiting for approval on every visual choice.
- Add a design-system page and a concise decision log for material product assumptions.
- Push commits and branches to this isolated repository so each iteration can be previewed.
- Use placeholder or synthetic assets when an approved Bridge asset is not present.

## Do not do

- Do not access, request, copy, or commit passwords, API keys, tokens, cookies, private documents, personal files, production records, client data, Supabase service-role credentials, AWS credentials, or real member information.
- Do not connect to Supabase, production APIs, authentication, email, payments, analytics, or any external database.
- Do not change `dillonmohr8777/bridge-discovery-prototype`.
- Do not delete or overwrite the original app, its staging previews, or its hosting configuration.
- Do not describe fictional profiles, metrics, or verification as real.
- Do not publish a production deployment or send client-facing communications from this design lab.

## Product boundaries to preserve

- Preserve the current MVP direction: directory, profiles, search and filters, onboarding direction, structured contact requests, business/license verification workflow, saved profiles, announcements, notifications, and admin moderation.
- Keep full social feeds, direct messaging, subscriptions/payments, native mobile apps, advanced recommendation AI, and advanced analytics out of scope unless separately approved.
- Keep the app accessible: semantic HTML, keyboard access, visible focus, readable contrast, reduced-motion support, meaningful headings, associated form errors, and no horizontal overflow.
- Keep all content clearly marked as a provisional design preview until Bridge branding and product decisions are formally approved.

## Definition of done for each iteration

1. Run `npm run typecheck`.
2. Run `npm run lint`.
3. Run `npm run build`.
4. Check desktop and phone widths, including keyboard focus and reduced motion.
5. Commit the work with a descriptive message.
6. Report the commit, changed routes, known assumptions, and the preview URL.

## Project boundary

The Pittsburgh concert venue, Blender environment, packed stadium, performer likeness, personal logo and imagery, and album-length performance concept belong exclusively to the **IMMOHRTAL website**. They are not a Bridge feature or a follow-up phase of Bridge. Do not plan, build, store, or publish any part of that concept in this repository.

Kimi's separate IMMOHRTAL assignment lives at `https://github.com/dillonmohr8777/immohrtal-kimi-redesign`. Kimi may work on both repositories, but Bridge changes stay here and all IMMOHRTAL/Blender work stays there.
